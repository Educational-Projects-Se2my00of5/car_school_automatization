Функциональные требования 3.0
1. Управление заданиями и критериями оценивания
1.1. Система должна предоставлять возможность создавать задание.
1.2. При создании или редактировании задания должна быть возможность добавлять один или несколько критериев оценивания.
1.3. Для каждого добавленного критерия система должна позволять задавать:

название критерия;

минимальный и максимальный балл (числовые значения);

комментарий к критерию (текстовое поле);

тип критерия: коэффициент (коэф), балл, ограничитель (например, оценка не больше 4);

настройку видимости (конфигурируемая видимость): для самих критериев и для оценок по ним – видны или не видны студентам.
1.4. Система должна позволять сделать критерии видимыми (в соответствии с настройками п. 1.3).

2. Расчет оценок
2.1. Оценка за задание должна рассчитываться как сумма всех полученных баллов по критериям, умноженная на все коэффициенты по критериям в пределах ограничителей по критериям.
2.2. Оценка за предмет должна рассчитываться как сумма по всем выбранным оценкам, где:

для оценок, на которые влияет коэффициент, берётся значение коэффициента контрольной, умноженное на соответствующий балл, при создании контрольной указываются список заданий на которые распространяется коэффициент контрольной;

для остальных оценок (не имеющих влияющего коэффициента) берётся просто балл.

3. Просмотр и отображение данных
3.1. Система должна предоставлять сводную таблицу, в которой для всех студентов видны:

список студентов;

поставленные им оценки по каждому критерию за задание.
3.2. Система должна предоставлять таблицу (журнал действий), в которой для каждой оценки по критерию видно, какой преподаватель её поставил и когда.
3.3. Система должна предоставлять таблицу, для каждого студента и задания и оценки за него.

4. Редактирование оценок
4.1. Преподаватели должны иметь возможность изменять оценки за критерии (баллы, коэффициенты, ограничители) в любое время после их выставления (в пределах допустимых значений, заданных в критерии).

5. Дедлайн
5.1. Дедлайн является отдельным полем при создании задания. Наказание за пропуск дедлайн задается тремя параметрами: Единица измерения времени(минута, час, день), числовое значение и значение ограничителя\коэффициента. Например, 10 мин, 0.1 балл тип ограничитель, это значит, что после наступления дедлайна максимальная оценка студента уменьшается на 0.1 балл за каждые 10 мин.



6. Работа критерия
6.1 Чтобы добавить критерий, преподаватель должен выбрать задание и нажать на него. После этого появится кнопка “добавить критерий”. После нажатия на кнопку открывается форма добавления критерия. Поля критерия описаны выше(1.3)
6.2 Чтобы добавить числовое значение определенному критерию,  преподаватель должен выбрать задание и нажать на него. После этого нажать на кнопку “Посмотреть все критерии”. После перехода, преподаватель может нажать на определенную ячейку и изменить\добавить значение.
6.3 Чтобы удалить критерию,  преподаватель должен выбрать задание и нажать на него. После нажатия на кнопку открывается форма добавления критерия. При нажатии правой кнопкой мыши на критерий, появится окно для удаления критерия.





Правила и обозначения 

1. Пользователи и роли (Users & Roles)
Пользователь (UserEntity) — человек, взаимодействующий с системой. Имеет: идентификатор (Uuid), имя, фамилию, контактные данные, список ролей.

Роль пользователя (UserRole) — интерфейс, определяющий тип доступа. Реализуется:

Менеджер (Manager) — управляет процессами (создание каналов, задач, метрик).

Студент (Student) — выполняет задания, получает оценки, участвует в командах.

Преподаватель (Teacher) — создаёт задания, проверяет работы, выставляет оценки.

Контактные данные (contactDetails) — строка с телефоном, email и т.п.

2. Задачи и типы заданий (Tasks)
Задача (Task) — учебное или организационное задание. Содержит:

идентификатор, метку (label), текст (text), список документов (documents).

тип задачи (TaskType) — влияет на процесс голосования и формирования команды:

Free — свободное формирование команд,

Last — последний вариант,

First — первый вариант,

Captain — капитанский выбор,

Democratic — демократическое голосование,

Qualified — квалифицированный (требует указания qualified_min).

тип команды (TeamType) — способ объединения студентов:

Random, Draft, Free.

минимальный размер команды (min_team_size).

дедлайн голосования (voting_deadline — optional).

минимальное количество квалифицированных участников (qualified_min — optional).

возможность перераспределения (is_can_redistribute).

алгоритм расчёта оценки (mark_algorithm — Sum или Medium).

Тип команды (TeamType) — перечисление: Random (случайная), Draft (драфт), Free (свободная).

Алгоритм оценки (MarkAlgorithm) — способ вычисления итоговой оценки: Sum (сумма), Medium (среднее).

3. Команды (Teams)
Команда (Team) — группа студентов, работающих над одной задачей. Характеристики:

идентификатор, название, ссылка на задачу (task_id).

капитан (captain_id) — необязательный; пользователь, выбранный капитаном.

флаги: isAvailableRevote (разрешён ли переголос), isCaptainVotingActive (активно ли голосование за капитана).

оценка команды (mark) — вычисляемое поле (может отсутствовать).

Голосование капитана (CaptainVote) — голос за кандидата в капитаны команды. Содержит:

идентификатор, team_id, user_id (кто голосует), candidate_id (за кого).

4. Голосование и оценивание
Голос (Vote) — запись о предпочтении пользователя при формировании команды. Связан с задачей (task_id) и пользователем (user_id).

Оценка (Marks) — индивидуальная оценка студента за задание. Содержит:

studentId, teacherId (кто выставил), taskId, числовое значение (mark), описание (необязательно).

Сдача работы (Submission) — попытка студента сдать задание. Поля:

идентификатор, user_id, task_id, необязательная оценка (mark), необязательный флаг isApproved (подтверждено ли преподавателем), список документов (идентификаторы).

5. Контент (каналы, посты, комментарии)
Канал (Channel) — контейнер для обсуждений (например, учебный курс). Имеет: идентификатор, метку, изображение, authorId.

Пост (Post) — сообщение в канале. Может быть типа:

Task (задание),

New (новость),

Control (контрольная точка).
Дополнительно: label, text, chanelId, authorId.

Комментарий (Comment) — ответ на пост. Поля: id, postId, text, authorId, признак редактирования isEdited.

Документ (DocumentEntity) — файл, прикреплённый к посту, задаче или сдаче. Тип документа (DocumentType): pdf, image, word (содержит путь).

6. Метрики и контролируемые значения
Метрика (Metric) — измеримый параметр задачи/поста. Атрибуты:

идентификатор, привязка к посту (postId) или задаче (taskId), имя, комментарий, минимальное/максимальное значение, тип метрики (MetricType), видимость (isVisible).

Тип метрики (MetricType) — определяет назначение:

Constraint (ограничение),

Coefficient (коэффициент),

Mark (оценка) — только в одном из вариантов enum.

Значение метрики (MetricValue) — фактическое значение метрики для конкретного пользователя. Содержит: id, value, userId (необязательно), metricId.

Изменение метрики (MetricChange) — аудит изменений значения метрики: id, время редактирования (edit_at), новое значение (edit_value), userId, metricValueId.

Контролируемый (Controlled) — сущность, привязанная к посту или задаче, возможно, для управления метриками.

7. Дедлайны (Deadline)
Дедлайн (Deadline) — временное ограничение для задачи. Поля:

taskId, тип времени (TimeType — не указан явно, предположительно абсолютное/относительное), числовое значение (timeValue), тип дедлайна (DeadlineType — не показан детально, вероятно, жёсткий/мягкий).

8. Важные бизнес-правила (на основе связей и комментария)
Оценка (Mark) рассчитывается при каждом новом запросе — т.е. не хранится статически, а вычисляется динамически на основе актуальных данных (например, результатов сдачи, метрик, алгоритма задачи). Значения полей mark в сущностях Team и Submission являются вычисляемыми (кэшируемыми) и могут пересчитываться при изменении входных данных.
Пользователь — Команда (студент может входить в несколько команд, команда содержит много студентов).

Функциональные требования 


1. Общие положения

Система должна обеспечивать возможность создания, управления и выполнения командных заданий в рамках курса. Основные роли:

Преподаватель – создаёт задания, управляет командами, устанавливает правила оценивания и дедлайны, может корректировать состав команд и оценки.

Студент – участвует в командах, сдаёт решения, участвует в голосовании (при необходимости), может предлагать перераспределение оценки.

2. Управление командами

2.1. Создание команд
Преподаватель может выбрать один из следующих режимов формирования команд для задания:

Рандомный – система автоматически распределяет студентов по командам (равномерно или с заданным размером).

Драфт (с выбором капитанов) – преподаватель вручную назначает капитанов, которые затем набирают команды (как в спортивном драфте).

Свободный – студенты самостоятельно объединяются в команды (как в Excel – сами выбирают, с кем быть). При конфликтах формирования преподаватель  вручную создает команды и перемещает студентов.

Для каждого режима должны быть предусмотрены настройки:

размер команды (фиксированный или диапазон),

возможность манипулировать количеством участников в команде 

2.2. Работа с командами для студентов
Студент видит свою команду, её состав, назначенного капитана (если есть), и может общаться с участниками (через внешние инструменты).

Если режим свободного формирования включён, студент может:

приглашать других студентов,
вступать в команду
покидать команду.

При свободном формировании количество доступных команд равно максимальному числу студентов посещающих предмет разделенных на минимальное количество людей в команде


3. Управление командными заданиями

3.1. Создание и настройка задания
Преподаватель может создать задание, указав:

название, описание,
сроки (дата/время начала, дедлайн сдачи, возможно, мягкий дедлайн),
тип сдачи решения (см. п. 4),
способ выбора капитана (см. п. 5),
правила оценивания (командная и персональная оценка, см. п. 6),
возможность перераспределения командной оценки (разрешено/запрещено, ограничения).

Задание может быть сохранено как шаблон для повторного использования в будущих курсах или в моменте (копирование).

3.2. Привязка к предмету
Каждое задание привязывается к конкретному курсу (предмету). У одного курса может быть несколько заданий, в том числе с разными составами команд.

3.3. Дедлайны
Система должна отслеживать дедлайны и блокировать прием решений после наступления срока. было бы хорошо, чтобы дедлайны не только блокировали возможность загрузки задания, но и “мягко намекали”, что пора начинать что-то сделать 

Преподаватель может продлить дедлайн для всей группы или для отдельной команды.

За какой то промежуток приходить оповещение о дедлайне задания


4. Типы сдачи решений

Для каждого задания преподаватель выбирает один из следующих режимов:

Тип
Описание
Любой член, но первое решение
Любой участник команды может загрузить решение, но проверяется только первый (все последующие игнорируются).
Любой член, но последнее проверяется
Несколько членов команды могут загружать варианты, но проверяется только последний (все предыдущие игнорируются).
Только капитан
Решение загружает только капитан команды.
Голосование – демократическое большинство
Каждый член команды голосует за один из вариантов решения (свой вариант голосовать нельзя). Выбирается вариант с наибольшим числом голосов. При равном количестве голосов – случайный выбор среди лидирующих.
Голосование – квалифицированное большинство
Требуется определённый процент голосов (например, 2/3) для принятия варианта. Если порог не достигнут, система может запросить доработку или предложить дополнительное голосование (по усмотрению преподавателя).



Для голосований:
Система должна вести учет, кто за какой вариант проголосовал.
Голосование может быть анонимным или открытым (настраивается преподавателем).
Срок голосования может быть ограничен (в рамках дедлайна).

5. Выбор капитана

Для задания может быть задан один из способов назначения капитана:

Выбирает команда – студенты внутри команды голосуют за капитана.

Назначает преподаватель – преподаватель указывает капитана для каждой команды (вручную или по правилу, например, первый в списке).

Капитан отсутствует – все члены равноправны.

Если капитан выбирается командой, система должна:

инициировать процесс голосования после формирования команды,
фиксировать результат.


6. Оценивание

6.1. Состав оценки
Каждое задание может давать студенту две составляющие оценки:

Командная оценка – единая для всех членов команды (выставляется преподавателем или рассчитывается автоматически по результатам сдачи).

Персональная оценка – индивидуальная надбавка или штраф (например, за активность, качество работы в команде).

Итоговая оценка студента может быть суммой или средним.

6.2. Командная оценка
Преподаватель может выставить командную оценку после проверки решения.

Командная оценка может быть перераспределена между участниками, если преподаватель разрешил такую возможность (см. п. 6.3).

6.3. Перераспределение командной оценки
Если преподаватель включил опцию «перераспределение оценки»:

Члены команды могут предложить новый коэффициент распределения (например, вклад каждого участника).
Перераспределение может быть инициировано:
по умолчанию после сдачи (если студенты договорились),
по запросу любого участника,
только капитаном.
Система должна проверять, что сумма коэффициентов равна 1 (или 100%).
Если перераспределение разрешено, итоговая оценка каждого = командная оценка × коэффициент.
Преподаватель может утвердить или отклонить предложенное распределение.

6.4. Персональная оценка
Преподаватель может вручную добавить/уменьшить персональную оценку для конкретного студента.


7. Функции для преподавателя

Создание/редактирование/копирование заданий (включая все параметры, перечисленные выше).

Просмотр списка команд по заданию, возможность редактирования состава в любой момент.

Просмотр и оценивание решений каждой команды.

Управление голосованиями (запуск, принудительное завершение, просмотр результатов).

Ручное вмешательство: изменение оценок, перевод студентов между командами, перенос дедлайна.

8. Функции для студента

Просмотр текущих заданий и информации о своей команде.

Сдача решения в соответствии с выбранным типом (загрузка файла, текст, ссылка).

Участие в голосованиях (за решение, за капитана).

Просмотр оценок (командной и персональной).

Предложение перераспределения оценки (если разрешено преподавателем).

Возможность покинуть команду (если режим свободного формирования).
